0. Whole System

整体心智模型

最重要的一句话是: GSD 不是几个 slash commands，而是一个运行在 Claude Code 上的工作流操作系统。 它把原本松散的一次次对话，变成一个有状态、可恢复、可分工的开发流水线。

为什么 GSD 有效

它把长对话拆成多个新上下文的子任务，尽量避免 context rot，同时把状态稳定地写回磁盘。

最容易误判的点

很多人以为核心逻辑在命令文件里。实际上命令层极薄，真正的 orchestrator 在 workflow 层。

教程的成功标准

不是“你知道仓库里有哪些文件”，而是“你能说清每个子系统的边界，并且自己搭出一套最小版 GSD”。

最小可复刻子系统

至少要有命令入口、.planning 状态层、init/query、discuss 产出 CONTEXT、plan 产出 PLAN、execute 产出 SUMMARY，以及一个 next 式状态推进器。

读后目标应该升级成：先复刻骨架，再补高级能力。 第一版先做 command → workflow → context → plan → execute → state update → next， AI-SPEC、UI-SPEC、Nyquist、bounce、seeds、hooks 全量护栏都可以后置。
复刻顺序 1
先做 .planning/ 和状态读写。没有状态层，你其实还没开始做 GSD。
复刻顺序 2
再做 init.* 这类 query，把文件系统事实收敛成 workflow 可消费的 JSON 快照。
复刻顺序 3
然后做 discuss-phase、plan-phase、execute-phase 这三个闭环核心。
复刻顺序 4
最后再补 next、milestone、hooks，以及 AI/UI/security 这些高级 gate。
1. Runtime Layout

Claude Code 安装形态

从 Claude Code 视角看，安装器的工作不是“复制几个 prompt”，而是把一整套命令、skill、agent、hook 和引擎映射到 .claude 运行时目录里。

.claude/commands/gsd/ Claude Code 的命令入口，本地项目里最常见。
.claude/skills/gsd-*/SKILL.md 全局安装时的现代 skill 入口，本质上是命令文件的重包装。
.claude/agents/ 可被 Task(subagent_type=...) 调用的角色卡。
.claude/get-shit-done/ workflow、references、templates、legacy CLI 等运行核心。
这也是为什么 GSD 看起来不像普通 prompt 工具。它必须同时占住命令入口、子代理入口、状态目录和 hook 生命周期。
2. Lifecycle

主流程全景: 从 new-project 到 new-milestone

之前那版图解最大的问题，是直接从 plan-phase 讲起了。真实的 GSD 主流程不是单条链，而是两个闭环：首次建项闭环和里程碑循环闭环。

为什么这张图重要

因为它解释了为什么仓库里不仅有 plan/execute，还一定会有 new-project、new-milestone、complete-milestone、next 这些看起来“偏管理”的命令。

/gsd-next 的真实定位

它不是神奇工作流，而是一个状态路由器。真正的状态跃迁还是靠 new-project、new-milestone、discuss-phase、plan-phase、execute-phase 这些主命令完成。

一个关键澄清

new-project 和 new-milestone 不是串联步骤，而是不同轮次的入口。首次建项走 new-project，只有当前 milestone 完成后，下一轮才进入 new-milestone。

3. Init vs Re-init

new-project 与 new-milestone

这两个命令共享一条骨架：收集目标、研究、requirements、roadmap、state 更新。但语义不同。一个是首次建项，一个是已有项目进入下一轮。

最实用的理解方式：new-project 负责把一个想法变成项目；new-milestone 负责把一个已有项目重新变成“下一轮可规划的项目”。
4. Decision Capture

discuss-phase 为什么是规划质量分水岭

这里是上一版最欠缺的部分。discuss-phase 不是“前置聊聊天”，而是把产品意图转成后续 research / planner 能直接消费的 locked decisions。

为什么它会直接影响 planning 质量

因为 planner 会同时读取 ROADMAP、REQUIREMENTS、RESEARCH、CONTEXT。少了 CONTEXT，planner 只能用默认判断；有了 CONTEXT，locked decisions 会直接进入 plan 约束。

怎么理解 --all / --auto / --chain

别把它们当参数表。它们本质上是在重新分配控制权：是先自动选灰区、自动选答案，还是只在 discuss 后自动接 plan/execute。

5. Planning Chain

图解 /gsd-plan-phase 1

这条链最能说明 GSD 的核心设计：命令层只做入口，workflow 做调度，planner / checker 才是真正的专业角色。

1
命令文件只是入口壳
commands/gsd/plan-phase.md 主要做三件事：声明工具、说明目标、把执行上下文指向 workflows/plan-phase.md。
2
workflow 先拉状态，不先拍脑袋
通过 gsd-remix-sdk query init.plan-phase 一次拿齐 phase 目录、模型、开关、现有工件、文件路径，再决定接下来分支。
3
planner 和 checker 职责彻底分离
planner 只负责把 phase 拆成可执行 PLAN，checker 只负责验证计划是否覆盖目标并能执行。
上面这张图只解决“谁调谁”。如果你要真正读透 /gsd-plan-phase，必须继续往下看 workflows/plan-phase.md 的 step 级逻辑，因为真正复杂的是 gate、revision loop 和状态写盘。
Step 0-3
初始化与参数规范化
先声明 planning 阶段不切分支，然后调用 init.plan-phase 一次取回 phase / config / file-path 快照。接着解析 --research、--skip-research、--gaps、--prd、 --reviews 等参数；如果 roadmap 里有 phase 但磁盘目录还没建，会在这里补齐。随后再校验 --reviews 前置条件和 roadmap phase 是否存在。
Step 3.5-5.7
planning 前置 gate
这里不是直接起 planner，而是连续过门：--prd 可以直接把 PRD 转成 CONTEXT.md；没有 CONTEXT.md 会提示先跑 discuss；AI 相关 phase 会检查 AI-SPEC.md；research 根据现有 RESEARCH.md 和 flags 决定是否复用；然后还有 Nyquist、security threat model、UI-SPEC、schema push 这些专门 gate。
Step 6-8
真正生成 PLAN 之前的上下文编排
如果已经有旧的 PLAN.md，workflow 会分流到追加 / 查看 / 重规划。之后把 STATE.md、ROADMAP.md、REQUIREMENTS.md、 CONTEXT.md、RESEARCH.md、REVIEWS.md、 UI-SPEC.md 等路径展开给 planner；如果没 PATTERNS.md，还会先起 gsd-pattern-mapper 去找代码库里的现成模式。
Step 8
planner 不是自由发挥，而是重约束生成
gsd-planner 拿到的不只是 phase 描述，而是完整输入包和一整套输出合同：每个 task 必须有 <read_first>、<acceptance_criteria>、 具体值写死的 <action>，frontmatter 里还必须带 wave、depends_on、files_modified、 autonomous。这一步本质上是在为 execute-phase 生产机器可消费的 plan。
Step 9-12.5
planner 返回后还有质量回路
planner 不只会返回 PLANNING COMPLETE，也可能返回 phase split 建议、source audit gaps、 checkpoint 或 inconclusive。之后 gsd-plan-checker 会验证 coverage 和可执行性；如果失败， workflow 会把 checker 的结构化问题回喂给 planner，最多跑 3 轮 revision loop。可选的 bounce 步骤还会把 plan 送去外部脚本 refinement， 但必须再次过 frontmatter 校验和 checker。
Step 13-15
收尾、写盘、自动推进
plans 通过后，还要再做 requirements coverage gate，确认 REQ-ID 和 CONTEXT 决策都被覆盖。然后更新 STATE.md、给 ROADMAP.md 标注 wave / cross-cutting constraints， 按配置提交 docs。若命中了 --auto / --chain / auto_advance， workflow 还会直接跳进 gsd-execute-phase。 
/gsd-plan-phase 1
  → commands/gsd/plan-phase.md
  → workflows/plan-phase.md
  → gsd-remix-sdk query init.plan-phase 1
  → Task(subagent_type="gsd-planner", ...)
  → Task(subagent_type="gsd-plan-checker", ...)
  → .planning/phases/01-*/01-01-PLAN.md
6. Execution Chain

图解 /gsd-execute-phase 1

执行阶段最像一个调度系统。它不直接写代码，而是用 phase-plan-index 发现哪些 plan 未完成、它们属于哪一 wave，再把每个 plan 发给 executor。

为什么必须有 phase-plan-index

因为执行阶段首先要知道“哪些 plan 已完成、哪些未完成、哪些可以并行”。这个查询把文件系统事实整理成 workflow 可消费的 JSON。

executor 真正负责什么

读单个 PLAN、逐 task 实现、跑验证、按 task commit、写 SUMMARY。它是实施工，不是调度器。

7. File Database

.planning/ 是文件型数据库

这层不是附带文档。它就是 GSD 的状态存储。Claude 读它，人也读它，git 也能追踪它。

PROJECT.md
回答“这个项目是什么”。这是最上层的产品语义。
ROADMAP.md
回答“它分成哪些 phase 来做”。是规划层的主轴。
STATE.md
回答“当前做到哪里、最近发生了什么、接下来该做什么”。是会话恢复的核心。
phases/*
记录某个 phase 从 context 到 research 到 plan 到 execute 到 verify 的完整生命周期。
8. Guard Rails

Hook 护栏系统

Claude Code 集成真正变强，不只是因为命令多，而是因为 GSD 把自己挂进了 settings.json 的生命周期事件里。

gsd-context-monitor.js

读取 statusline 产出的上下文指标，低于阈值时用 additionalContext 提醒 agent “别再开新大活了”。

gsd-prompt-guard.js

在写入 .planning/ 前扫描 prompt injection 模式，作为 advisory-only 预警，不强行阻断流程。

9. Programmatic Core

SDK / Query / Runner 这一层在干什么

当你看到 workflow 里反复出现 gsd-remix-sdk query ...，本质上是在调用一层面向 orchestration 的 API，而不是直接读文件。

sdk/src/query/index.ts
注册大量 query handler。这里像一个内部 API 注册表，负责 state、phase、roadmap、init、commit 等操作。
sdk/src/query/init.ts
最关键的一层。把多个底层查询组合成 init.plan-phase、init.execute-phase 这种“启动快照”。
sdk/src/gsd-tools.ts
工具桥。上层 runner 不必关心当前命令是走 SDK registry 还是 fallback 到 legacy CJS。
sdk/src/phase-runner.ts
把 discuss → research → plan → execute → verify 这条生命周期程序化成显式状态机。
一句话总结：workflow 是 prompt 级 orchestrator，runner 是代码级 orchestrator，query 层是两者共享的数据接口。
